The Legend of Ötzi the Iceman — Customizable Control Scheme

Design target: portrait-first mobile controls for The Legend of Ötzi the Iceman, a top-down Canvas 2D survival adventure. The player explores Copper Age forests, alpine passes, caves, river bends, hunting grounds, and rival trails while managing warmth, hunger, stamina, tools, and carried resources.

portrait-first customizable safe-area aware Pointer Events burst dodge/sprint Canvas 2D Hermes/Codex-safe

1. Core recommendation

The control scheme should support a slower, survival-weighted adventure instead of a generic dual-stick shooter. Ötzi needs to move, aim, gather, craft, dodge, open the trail map, and use tools without forcing the player to hold fragile buttons while steering with two thumbs.

Default layout

Top: large top-down play window. Below: compact stat band, sliding trail-map tab, and a dedicated control deck. This keeps the wilderness readable while still giving the thumbs stable controls.

MOVE stays bottom-left

The left thumb owns walking, sneaking, and river/cave movement. It should never share space with the minimap, craft menu, dodge, or use/gather action.

AIM/LOOK becomes right-side strip

When no ranged weapon is equipped, this strip controls facing/look direction. When the bow or thrown stone is equipped, it becomes an aim surface.

Actions are explicit buttons

USE/GATHER, DODGE/SPRINT, and CRAFT/MENU need large buttons. Sprint should be a burst/toggle so the player can evade, aim, or gather without awkward thumb holds.

4. Customization model

Customization should be declarative. The engine reads a saved profile, lays out controls from data, and writes changes back through CRAFT/MENU → OPTIONS → EDIT CONTROLS.

5. Edit Controls flow

  1. Player opens CRAFT/MENU → OPTIONS → EDIT CONTROLS.
  2. The options overlay dims or collapses so it does not block controls being edited.
  3. Each control shows its boundary, label, resize handles, and reset button.
  4. Player drags controls inside legal control deck / map band boundaries.
  5. Player uses + and − to resize the selected control.
  6. Player taps Save, Reset Portrait, Reset All, Left-Handed Mirror, or Export Profile.

The editor must clamp controls away from browser gesture edges and safe-area cutouts. It should never allow a critical survival button to sit under Android navigation, iOS home indicator, a notch, or the browser back gesture edge.

6. Runtime input rules

Pointer ownership

Each active pointer gets one owner: move, aim, use, dodge, craft/menu, map, or UI. Do not let one pointer trigger two gameplay controls.

Pointer capture

When a pointer starts on MOVE or AIM, capture it until pointerup or pointercancel. Dragging off the visible control should still work.

Cancel safety

On pointercancel, blur, orientation change, pause, cutscene, dialogue, or scene transition, zero movement and aim input immediately.

Action sampling

DOM events update raw pointer/key state only. The fixed-step game loop samples the action map once per tick.

Input pipeline

Pointer/keyboard/gamepad events
        ↓
INPUT module stores raw state by pointerId/key
        ↓
CONTROL_LAYOUT assigns raw input to control zones
        ↓
ACTION_MAP emits move/aim/use/dodge/craft/map
        ↓
GAME UPDATE consumes actions
        ↓
RENDER draws world, survival HUD, control feedback, and debug overlay

7. Implementation notes for Ötzi

  • Keep controls declarative. Do not hardcode button coordinates in multiple modules.
  • Use CRAFT/MENU as the phone pause path. Short press opens quick craft; long press opens pause/options/inventory/facts.
  • Keep trail map independent. Trail map lives in its own tab/band, not inside MOVE or AIM zones.
  • Keep dodge forgiving. Recommended default: tap for a short dodge, double tap for sprint toggle, stamina drains normally.
  • Keep AIM context-aware. No bow equipped = face/look; bow equipped = aim; trap equipped = placement direction.
  • Keep safe-area math centralized. Resize/orientation changes rebuild the layout from the profile.
  • Do not couple visual size to gameplay strength. Making DODGE bigger should not change dodge distance or stamina cost.
  • Do not let dialogue eat movement state. Opening a fact dialogue must zero raw movement/aim to prevent stuck walking.

11. Final design contract

Ötzi controls should feel like a survival tool belt attached to a top-down wilderness screen. The world remains readable. The survival stats remain visible. The trail map has a stable home. MOVE, AIM, USE, DODGE, and CRAFT/MENU are customizable but still bound by a layout contract that protects mobile play from regression.
