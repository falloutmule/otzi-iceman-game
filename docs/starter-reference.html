1. Live starter demo

This demo is intentionally small but structured like the engine should be structured. It generates its own tile atlas, sprite atlas, world map, resource nodes, NPCs, particles, fog/vignette, minimap, collision overlay, and save/export state. Codex can pull the same module names into split files later.

What this demo proves

  • No image files: tiles and sprites are Canvas-generated at startup.
  • Seeded generation: same seed makes the same forest/cave/resource layout.
  • Game-loop shape: requestAnimationFrame drives a fixed-step update accumulator.
  • Separation: render caches, simulation state, input state, save state, and debug state are separate.
  • Mobile pattern: Pointer Events drive a left stick and large right buttons.
  • Starter, not final game: this is a Codex launchpad with deliberate extension points.

The live code is embedded near the bottom under Source map. It is readable on purpose. Do not minify until release.

2. Final direction

Build custom Canvas 2D

Use a custom engine. Phaser-like scene and asset-cache ideas are worth copying conceptually, but not the runtime dependency. The single-file/no-external-assets target makes a small custom loop easier for AI agents to inspect and repair.

Canvas world, DOM panels

Draw world, entities, particles, overlays, minimap, and debug collision on Canvas. Use DOM/CSS only for pause menu, inventory, dialogue panels, settings, and documentation overlays.

Procedural assets first

Start with generated low-color graphics: ground tiles, trees, rocks, deer, campfire, buildings, icons, silhouettes, fog, warmth vignette, and UI tokens. Embedded art can come later only if it stays small.

Data-driven history

Facts about Ötzi should live in a fact database with source fields. Game text can separate verified archaeology, plausible gameplay interpretation, and fictionalized myth/ending.

3. AI-safe source constitution

Paste this at the top of the eventual game source. It tells Codex/Hermes how to avoid breaking the engine.

/*
OTZI CANVAS 2D ENGINE CONSTITUTION

GOAL
- Strict single-file HTML5 game for final release.
- Development may be split into src/*.js, but dist/index.html is the playable artifact.
- No external runtime assets, no CDN, no network calls, no external fonts.

ENGINE SHAPE
- Canvas 2D renders the world.
- DOM/CSS renders menus, settings, inventory, dialogue, and debug panels.
- The game is mobile portrait first, Android Chrome first, iOS Safari fallback.
- Every visual asset needed for the MVP must be procedural or embedded as tiny data.

SECTION OWNERSHIP
- CONFIG owns constants only.
- RNG owns deterministic random generation only.
- VIEWPORT owns CSS size, DPR, backing buffer, safe areas, and visualViewport handling.
- INPUT owns raw keyboard/pointer/gamepad state only.
- ACTION_MAP converts raw input to semantic actions.
- SAVE owns versioned persistent state and migrations.
- AUDIO owns one AudioContext and procedural sound functions.
- ASSETS owns generated atlases/canvases. It does not mutate game state.
- TILEMAP owns map cells, semantic layers, chunk data, and collision flags.
- WORLDGEN owns seed-based generation and solvability checks.
- ENTITIES owns entity arrays and behavior dispatch.
- COLLISION owns movement resolution and tile/entity collision helpers.
- CAMERA owns world-to-screen conversion and camera smoothing.
- SCENES owns scene transitions only.
- DIALOGUE owns text queues and fact display.
- INVENTORY owns items, stack counts, equipment, and caps.
- CRAFTING owns recipes and validation.
- VILLAGE owns unlocked building functions, not complex colony AI.
- RENDER_WORLD draws world and entities. It must not mutate simulation state.
- RENDER_UI draws Canvas HUD/minimap/debug. DOM UI is updated separately.
- DEBUG owns overlays, seed display, test hooks, and benchmark mode.
- TEST_HOOKS exposes stable functions for Playwright/Codex tests.

MANDATORY INVARIANTS
- Same engineVersion + same seed + same procedural inputs = same generated world.
- Saves store seed plus deltas, not transient render caches.
- No save schema change without bumping SAVE_VERSION and adding migration notes.
- No visual change without screenshot proof.
- No mobile input change without pointercancel/rotation/resize testing.
- No render-loop allocation spikes: avoid creating objects inside hot loops.
- Do not edit minified release output during development.
*/

5. Procedural graphics library

The starter engine should use a small, deterministic procedural asset system. Each recipe draws into an offscreen canvas once, then the render loop uses drawImage from the cache. The recipes below are intentionally simple: mostly fillRect, arc, lineTo, globalAlpha, and seeded jitter.

// 3. Never draw procedural noise directly inside the main render loop.
// 4. Never load external PNG/SVG/fonts for MVP.

6. Tilemap and world model

Recommended: a chunked typed-array tilemap for generated worlds, plus object/entity layers for gameplay. Do not store everything as objects. Do not store everything as one giant string once you need mutation. Use compact typed layers in memory, and save only seed plus deltas.

LayerTypeOwnsSave behavior
groundUint16Arraygrass, dirt, snow, stone, water, cave floor.Generated from seed; save changed tiles only.
objectUint16Arraytree, boulder, flint scar, bush, door, chest, workstation.Generated from seed; save harvested/removed/built deltas.
flagsUint16Arrayblocked, water, warm, cold, trigger, harvestable, discovered.Derived + deltas for fog-of-war and mutations.
biomeUint8Arrayforest, alpine, river, cave, village, rival trail.Generated from seed; save region id only.

8. Survival, crafting, and village loop

Keep survival readable. Each stat should change player decisions, not become a spreadsheet. Village systems are function unlocks, not simulated pawns.

Survival stats

  • Health: fail state and injury pressure.
  • Stamina: sprint/dodge/combat budget.
  • Hunger: slow pressure; affects stamina regen.
  • Warmth: cold-region timer; fire/clothing matter.
  • Wetness: increases cold drain; river/rain cost.
  • Weight: limits resource hoarding.

Crafting scope

  • Stick + flint + fiber → crude tool.
  • Birch bark + fiber → container.
  • Tinder fungus + stick → fire kit.
  • Hide + grass → warmth upgrade.
  • Flint + shaft + feather/fiber → arrow bundle.

Dungeon unlocks

  • Cave/flint scar → passive flint gathering.
  • Birch grove → passive bark/fiber gathering.
  • Hunting ground → food/hide access.
  • River bend → fish/water route.
  • Rival trail camp → story/final conflict route.

Village functions

  • Trade hut: convert surplus.
  • Healer/fire: restore health/warmth.
  • Workshop: unlock upgrades.
  • Storehouse: raise caps.
  • Fact NPCs: teach verified archaeology.

10. Save/load schema

Versioned saves should be boring and durable. Use localStorage through try/catch; always include export/import string fallback. Under file://, browsers may treat storage differently, so the game should degrade to export/import. 

const SAVE_VERSION = 1;

const saveShape = {

11. Procedural audio

Use one AudioContext, unlocked by a Start button. Sounds are short oscillator/noise envelopes: footstep, gather, craft, bow draw/release, arrow hit, deer flee, rival alert, cold wind, fire crackle, UI click, village unlock, low-health warning. 

// Procedural sound rule:
// - Create/resume AudioContext only from user gesture.
// - Use gain ramps; do not snap gain from 1 to 0.
// - Keep every sound optional behind mute/volume settings.
// - Avoid autoplay assumptions.
// - Embedded samples are optional later, but not required for MVP.

// Playwright-style smoke test sketch
test("Otzi starter loads and moves", async ({ page }) => {
  const requests = [];
  page.on("request", r => requests.push(r.url()));
  const errors = [];
  page.on("pageerror", e => errors.push(e.message));
  page.on("console", msg => { if (msg.type() === "error") errors.push(msg.text()); });

  await page.goto("http://127.0.0.1:8080/dist/index.html");
  await expect(page.locator("canvas#worldCanvas")).toBeVisible();

  await page.click("#startBtn");
  await page.keyboard.down("KeyD");
  await page.waitForTimeout(500);
  await page.keyboard.up("KeyD");

  const snap = await page.evaluate(() => window.__OTZI_TEST__.snapshot());
  expect(snap.player.x).toBeGreaterThan(0);
  expect(errors).toEqual([]);
  expect(requests.filter(url => !url.startsWith("http://127.0.0.1:8080"))).toEqual([]);

  await page.screenshot({ path: "proof/otzi-starter-reference-seed.png" });
});

13. Codex implementation cards

Card 2 — Improve procedural tiles

Add biome palettes, transition tiles, river edges, cave floors, snow patches, and generated minimap icons. Produce atlas screenshot proof.

Card 3 — Resource gathering loop

Implement stick/stone/bark/grass/flint nodes, gather timing, inventory caps, node depletion deltas, and save/load round trip.

Card 4 — Dungeon run

Add cave map generator, guaranteed entrance/exit path, flint objective, completion flag, and village unlock.

Card 5 — NPC fact dialogue

Add facts database with source fields and verified/plausible/fictional labels. NPCs reveal facts and mark discovered facts in save.

Card 6 — Mobile proof

Run Android Chrome test: first load, Start/audio, joystick+use multitouch, pointercancel, orientation, resize, save/reload, screenshot evidence.
